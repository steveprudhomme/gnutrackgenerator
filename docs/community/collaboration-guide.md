# Guide de collaboration communautaire

GNU TrackGenerator est ouvert à la collaboration. Ce guide explique comment participer sans devoir connaître tout le projet dès le départ.

## Ce que signifie “ouvert à la collaboration”

Le projet accueille :

- les utilisateurs qui posent de bonnes questions ;
- les personnes qui reproduisent des bogues ;
- les rédacteurs et rédactrices de documentation ;
- les traducteurs et traductrices ;
- les designers ;
- les développeurs et développeuses ;
- les personnes qui améliorent les tests ;
- les personnes qui aident à trier les tickets ;
- les personnes qui proposent des idées musicales ou pédagogiques.

La collaboration ne signifie pas que toutes les demandes seront acceptées immédiatement. Elle signifie que les règles d’accueil, de discussion, de contribution et de décision sont publiques.

## Première contribution suggérée

1. Lire `README.md` pour comprendre la mission.
2. Lire `CODE_OF_CONDUCT.md` pour connaître les attentes communautaires.
3. Lire `CONTRIBUTING.md` pour connaître le processus technique et le DCO.
4. Choisir un ticket simple.
5. Poser une question si le ticket est ambigu.
6. Proposer une demande de tirage courte.

## Bons premiers tickets

Les bons premiers tickets devraient être :

- clairement décrits ;
- limités ;
- non bloquants ;
- accompagnés de critères d’acceptation ;
- utiles pour apprendre la structure du projet.

Étiquettes recommandées :

- `good first issue` ;
- `help wanted` ;
- `documentation` ;
- `test` ;
- `accessibility` ;
- `translation` ;
- `triage`.

## Comment proposer une idée

Une bonne proposition décrit d’abord le problème :

```text
Comme [type d’utilisateur], je veux [capacité] afin de [résultat utile].
```

Puis elle précise :

- le contexte ;
- un exemple concret ;
- les alternatives envisagées ;
- l’impact possible sur les utilisateurs existants ;
- le degré d’urgence réel.

## Comment faire une revue constructive

Une revue utile :

- distingue les blocages des préférences ;
- explique le pourquoi ;
- propose une alternative ;
- remercie la personne pour son travail ;
- reste centrée sur le changement proposé.

Exemples de formulations :

```text
Bloquant : ce changement casse le format existant. Peut-on ajouter une migration ?
```

```text
Suggestion non bloquante : ce nom pourrait être plus explicite.
```

## Comment demander de l’aide

Une bonne demande d’aide inclut :

- ce que vous essayez de faire ;
- ce que vous avez déjà tenté ;
- le résultat attendu ;
- le résultat observé ;
- la version utilisée ;
- les logs pertinents, sans secrets.

## Limites de la collaboration

Le projet ne peut pas garantir :

- une réponse immédiate ;
- l’acceptation de toute fonctionnalité ;
- la maintenance d’intégrations non documentées ;
- le support privé gratuit ;
- la compatibilité avec tous les environnements.

Ces limites servent à protéger la qualité du projet et la santé des mainteneurs.

## Reconnaissance

Les contributions peuvent être reconnues dans :

- l’historique Git ;
- `CONTRIBUTORS.md` ;
- les notes de version ;
- les discussions publiques liées au changement.

Une personne peut demander à ne pas être nommée publiquement.
