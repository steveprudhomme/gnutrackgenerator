# Politique de support

Ce document définit les attentes de support afin de protéger le temps des mainteneurs et de rendre l’aide communautaire plus prévisible.

Le projet est libre et gratuit. Cela ne crée pas une obligation de support individuel, immédiat ou garanti.

## Aide communautaire

Le support communautaire est encouragé : une personne utilisatrice peut répondre à une autre, proposer un exemple, améliorer la documentation ou confirmer qu’un bogue est reproductible. Les mainteneurs ne sont pas les seuls points d’aide du projet.

## Où demander de l’aide

| Type de demande | Canal recommandé |
|---|---|
| Question d’utilisation | Forum ou discussions communautaires : `[lien à ajouter]` |
| Question rapide entre utilisateurs | Salon de clavardage : `[lien à ajouter]` |
| Documentation manquante | Ticket de documentation |
| Bogue reproductible | Ticket de bogue avec modèle complet |
| Vulnérabilité | Signalement privé selon `SECURITY.md` |
| Demande commerciale ou institutionnelle | Contact mainteneurs : `[adresse à ajouter]` |

## Questions d’utilisation

Les questions d’utilisation incluent notamment :

- comment installer ou lancer le projet ;
- comment configurer un environnement local ;
- comment interpréter un message d’erreur courant ;
- comment choisir une option ;
- comment accomplir une tâche avec le logiciel.

Ces questions doivent être posées dans les espaces communautaires, pas directement aux mainteneurs individuellement, sauf invitation explicite.

## Bogues avérés

Un bogue doit être soumis comme ticket public seulement si vous pouvez fournir :

- le comportement attendu ;
- le comportement observé ;
- les étapes précises de reproduction ;
- la version utilisée ;
- l’environnement pertinent ;
- les journaux ou captures nécessaires ;
- une réduction minimale du problème si possible.

Un ticket incomplet peut être fermé ou converti en discussion jusqu’à ce qu’il soit reproductible.

## Délais de réponse

Les mainteneurs font un effort raisonnable pour répondre, mais aucun délai n’est garanti.

Attentes réalistes :

- Les questions communautaires peuvent recevoir une réponse quand une personne disponible les voit.
- Les bogues bien documentés sont priorisés selon leur gravité, leur reproductibilité et leur impact.
- Les demandes vagues, urgentes seulement pour une organisation externe ou impossibles à reproduire peuvent ne pas être priorisées.
- Les vulnérabilités suivent la procédure de `SECURITY.md`.

## Ce que le projet ne fournit pas par défaut

Le projet ne fournit pas gratuitement et automatiquement :

- support privé individuel ;
- assistance en temps réel ;
- intégration sur mesure ;
- correction prioritaire pour un cas d’affaires spécifique ;
- formation personnalisée ;
- audit de sécurité complet ;
- garantie de compatibilité avec tous les environnements.

## Comment obtenir une meilleure aide

Pour maximiser les chances d’obtenir une réponse utile :

1. Lire le README et la documentation pertinente.
2. Chercher dans les tickets existants.
3. Fournir un exemple minimal reproductible.
4. Décrire ce que vous avez déjà essayé.
5. Joindre les journaux utiles en retirant les secrets.
6. Rester courtois et patient.

## Respect du temps des mainteneurs

Les mainteneurs peuvent fermer une demande si elle :

- ne respecte pas les modèles fournis ;
- demande du support personnalisé sans information suffisante ;
- duplique une discussion existante ;
- contient des propos irrespectueux ;
- contourne la procédure de sécurité ;
- exige une réponse immédiate.

Fermer une demande n’est pas un jugement sur la personne. C’est un mécanisme de tri nécessaire à la viabilité du projet.
